## I. Derived Asset Analysis and Information Writing

## Input/Output Rules

1. **Input Data**: Read `<screenplay>` and `<assets_database>` from the project workspace.
2. **File Persistence**: Save your final derived assets directly to the project workspace using file-writing tools. Follow the global path rules defined in `AGENTS.md`.

### Execution Process

1.  Analyze the `<screenplay>` and `<assets_database>` files.
2.  **Directly analyze the script and asset descriptions**, autonomously determining if each asset has visual state variations requiring derivation (do not read or rely on director's plans/pre-planned lists).
3.  Identify derivations asset by asset according to the "Extraction Rules" below: **roles only extract transformation states, scenes only extract time variations, props do not extract any variations**.
4.  For each identified derivation, generate complete `name`/`desc`/`type` according to the field rules below.
5.  Briefly explain the newly added derived asset content (within 200 characters).
6.  If no assets require derivation, return "No derived assets needed", and the process ends.
7.  **Write each new derived asset** to the assets markdown file (create a new unique ID when adding new, and completely fill `assetsId`/`name`/`desc`/`type`).
8.  After all calls are completed, return a brief confirmation (e.g., "Derived assets written, N entries in total").

### Mandatory Constraints (Prevent Missing Calls / Overstepping Authority)

-   **Strictly adhere to the extraction scope**: Roles are limited to transformation states (costumes / transformation effects / deformation), scenes are limited to time variations, props are never derived; states outside this scope must not be written.
-   After identifying derived assets, you must actually write them to the assets markdown file; only outputting analysis text is considered an incomplete task.
-   Do not return "completed" type results if the writing task has not been completed.

### Derived Asset Data Structure Requirements
When writing to the assets markdown file, use the following structure:
```json
{
"assetsId": number, //Associated asset ID
"id": number, // Derivative asset ID, a unique ID is generated when adding a new asset
"name": "string", // Derivative asset name
"desc": "string", // Derivative asset description
"type": "role" | "scene" // Derivative asset type
}
```

Field description:
-   `assetsId`: The ID of the parent asset in the workspace.
-   `id`: Must be `null` when adding new; fill in the existing derived asset ID when updating an existing derived asset.
-   `name`: 2-6 characters, reflecting visual appearance changes.
-   `desc`: `[Differences from default state] · [Visual characteristics]`, 1-100 characters.
-   `type`:
    -   Fill `role` for character derivations.
    -   Fill `scene` for scene derivations.
    -   Props are not derived in this stage, so `tool` will not be generated; `clip` is only used for shot/clip-level assets and does not normally appear.

### Extraction Rules

> **Core Principle**: A derive is a **visual state variation** of the parent asset ('{Parent Asset Name}·{State Name}'), **not** an independent object, nor a temporary close-up extracted for a specific shot.
> **Autonomous Determination at this Stage**: Whether derivation is needed is determined directly by this stage based on the script and asset descriptions, without reading director's plans or relying on any pre-planned lists.
> **Character Baseline State**: The character's parent asset defaults to the basic attire corresponding to the character's identity (generated by `art_character.md` based on the character description). Transformation/costume change derivations are implemented according to `art_character_derivative.md` for the corresponding style.
> **Scene Baseline State**: The scene's parent asset defaults to the basic time-of-day view of that scene (generated by `art_scene.md`). Time variation derivations are implemented according to `art_scene_derivative.md` for the corresponding style, using a 'reference main view + target time period' approach.

**Extraction Scope (by Asset Type)**:

| Asset Type | Derivable? | Extraction Scope | Examples |
|------------|------------|------------------|----------|
| Character  | Yes        | **Only transformation states**: ①Costumes; ②Transformation effects; ③Deformation | School uniform→Battle suit/Gown, Transformation light effects/Energy entanglement, Bestial transformation/Gigantification/Missing limbs |
| Scene      | Yes        | **Only time variations** | Daytime→Nighttime, Dusk version, Dawn version |
| Prop       | No         | Do not extract any variations | — |

**Rules**:
-   Only extract states that have a clear visual difference from the default state and cannot be controlled by prompts alone for the model.
-   **Characters**: Only extract "transformation state" derivations, in three directions——①**Costumes** (overall change in attire/dress, e.g., school uniform→battle suit, gown, armor); ②**Transformation effects** (light effects, energy, particles, etc., during transformation or form switching); ③**Deformation** (changes in body shape, structure, overall form, e.g., bestial transformation, gigantification, mutation, missing limbs). These three types can coexist.
-   **Scenes**: Only extract "time variations"——overall lighting/color tone/atmosphere changes for the same scene at different times of day (e.g., daytime→nighttime, dusk, dawn). A single scene can have multiple time variations, each independent; other changes such as angle, weather, destruction, etc., are **not extracted** in this stage.
-   **Props**: Never extract any derivations.
-   Character transformation/deformation feature variations must simultaneously meet: **stable, reusable, asset-level**. Only create when they persist across multiple shots/scenes and alter the character's overall identifiable appearance.
-   The following situations **never require derivation**: close-ups of specific body parts like the back of a hand/eyes/lips; instantaneous expressions or emotional states such as 'terrified face' or 'red-rimmed eyes'; local textures that can be expressed by storyboard descriptions or prompts; still frames made for horror hooks or emotional emphasis in a single shot.
-   **Common Misjudgment Reason**: Mistaking "script's key description" for "requiring a derived asset". The criterion is not its importance, but whether it belongs to the parent asset's **stable, reusable, overall-level** visual state.
-   Only add corresponding derivations when a character in the script clearly undergoes a costume change/transformation/form alteration; if the basic attire is maintained throughout and there is no transformation or form change, then no derivation is made.
-   Do not duplicate states already present in the `derive` array.
-   1-5 derivations per asset, quality over quantity.
-   After extracting derived assets, you must write each entry to the assets markdown file to save it; analyzing without writing is prohibited.
-   Source Priority: Explicit script description > Asset description implication > Reasonable inference.
-   `name`: 2-6 characters, reflecting visual appearance changes.
-   `desc`: Format is `[Differences from default state] · [Visual characteristics]`.

## 2. Derivative Asset Image Generation

## Input/Output Rules

1. **Input Data**: Read `<derived_assets>` from the project workspace.
2. **Action Execution**: Use the `generate_image` tool to generate images based on the descriptions, and output the resulting mappings to the workspace following `AGENTS.md`.

### Execution Flow

1.  Read the assets markdown file and collect all asset IDs for which images need to be generated.
2.  Use the `generate_image` tool to generate images based on the asset descriptions and save their paths to the assets markdown file.

### Constraints

- Prerequisite: Derivative asset analysis has been completed and written.
- Initiate generation only for assets that have a derivative status and have not yet had images generated.
